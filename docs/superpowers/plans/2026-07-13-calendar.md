# Calendar Section Implementation Plan

**Goal:** A dedicated Calendar page with Day/Week/Month views over Google + Apple events, per-team calendar selection, hide event/series, a second time zone in Week; then event CRUD + Google Meet; then coworker overlay + invitee availability.

**Architecture:** New Inertia page `GET /{current_team}/calendar` → `CalendarController` → `pages/calendar/Index.vue`. Custom Vue grid components on `date-fns`/`core/dates.ts`. Reuse `eventPrefs` (per-team) for calendar visibility + hidden events. Google via the existing raw-`Http` proxy pattern (no `google/apiclient`); scopes expand only in Phase 2.

**Tech:** Laravel 13, Inertia v3, Vue 3, date-fns, Tailwind 4, Pest, Vitest.

## Global constraints
- Do not reuse `currentView.kind === 'calendar'` (that means a period *note*). The events page is its own Inertia page.
- Google account data is user-global; calendar *visibility* is per-team (client-side `eventPrefs`, keyed by teamSlug).
- Phase 2 requires OAuth scope `calendar.events` (+ `calendar.readonly`) and user re-consent.
- Apple stays read-only (native bridge).

---

## Phase 1 — Calendar page + views (no new Google scope)

- **Task 1 — Route + page shell.** `CalendarController` (Inertia render `calendar/Index`, props: team slug, googleConnected, members). Route in `{current_team}` group. Sidebar: add a "Calendar" `<Link>`; relabel the period-notes section header "Calendar" → "Journal". Test: `CalendarPageRenderTest` (renders, component `calendar/Index`).
- **Task 2 — Richer events endpoint.** Extend `FetchGoogleEvents` (or a new `CalendarEventsController`) to accept an arbitrary `start`/`end` range, paginate (`pageToken`), and return `{id, calendar_id, calendar_name, color, all_day, start, end, location, html_link, hangout_link, organizer_email, attendees[], recurring_event_id, status}`. Keep the 5-min cache keyed by range. Move the events route under the team prefix (or keep global + pass team). Test: feature test with `Http::fake` asserting pagination + shape.
- **Task 3 — Client calendar store.** `resources/js/stores/calendar.ts`: current view kind (day|week|month) + anchor date, navigation (prev/next/today), a fetch that pulls Google (HTTP) + Apple (bridge) for the visible range into a `CalendarEvent[]`, and derived layout helpers. Reuse `eventPrefs` for visibility/hidden. Vitest for the pure range→columns/positioning math (`core/calendarLayout.ts`).
- **Task 4 — Week view.** 7 day-columns + hour grid, all-day row, event blocks positioned by start/end (overlap columns), current-time line, and a **second time-zone** left axis (pref-selected). Vitest covers the layout function; component wired to the store.
- **Task 5 — Day view.** Single-day version of the week grid (reuse layout).
- **Task 6 — Month view.** 6-week grid, events as chips per day, "+N more" overflow, click a day → Day view.
- **Task 7 — Calendar picker + hide.** A panel listing Google + Apple calendars with per-team show/hide (eventPrefs), and event/series hide reusing `hiddenEvents`. Second-time-zone picker stored in a pref.
- **Task 8 — Verify Phase 1.** pint/larastan/tsc/eslint/build + full suites; browser smoke (author's own events render across views).

## Phase 2 — Event CRUD + Google Meet
- Expand OAuth scopes (`calendar.events`), re-consent flow + messaging.
- Write endpoints: create/update/delete event (conferenceData/`conferenceDataVersion=1` for Meet); server actions with `Http`.
- Event editor dialog (title, time, all-day, calendar, description, location, attendees, Meet toggle); drag/click-to-create in Week/Day.
- Tests: `Http::fake` for insert/patch/delete + Meet payload; policy (only the connected user's calendars).

## Phase 3 — Coworker overlay + availability
- Add-person-by-email overlay. **Prefer full event details**: call `events.list`
  on the colleague's calendar (their email as the calendar id) with the current
  user's token — this returns real events (title, description, attendees) when
  their calendar is shared at "see all event details" (the common Workspace
  internal default). If that returns 403/limited, **fall back to FreeBusy**
  busy-blocks (opaque). Surface which mode is in effect per person.
  Rationale: the user previews colleagues' actual meetings to judge importance
  and deliberately overlaps low-priority ones — free/busy alone is insufficient.
- FreeBusy endpoint (`/freebusy`, items[]≤50) used both for the fallback and for
  the invitee availability grid in the event editor.
- Merge overlay lanes into Week/Day; distinct styling per person; a per-event
  detail popover (name, description, attendees) for overlaid events.
